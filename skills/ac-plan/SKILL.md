# Plan - 单模型规划

$ARGUMENTS

---

## 核心协议

- **语言协议**：与工具/模型交互用**英语**，与用户交互用**中文**
- **单模型模式**：仅允许单模型分析与规划，禁止多模型分工、交叉验证与并行协作描述
- **上下文检索**：统一指定使用 `ace-tool` 进行语义搜索
- **代码主权**：外部模型对文件系统**零写入权限**，所有修改由 Claude 执行
- **止损机制**：当前阶段输出通过验证前，不进入下一阶段
- **仅规划**：本命令允许读取上下文与写入 `.claude/plan/*` 计划文件，但**禁止修改产品代码**

---

## 规划约束

**工作目录**：
- `{{WORKDIR}}`：使用当前工作目录的绝对路径作为检索与计划输出根目录
- 如果用户通过 `/add-dir` 添加了多个工作区，先用 Glob/Grep 确定任务相关的工作区
- 如果无法确定，用 `AskUserQuestion` 询问用户选择目标工作区

**角色提示词**：

| 阶段 | 文件 |
|------|------|
| 分析 | `./analyzer.md` |
| 规划 | `./architect.md` |

**重要**：
- 只允许生成和更新 `.claude/plan/*.md` 计划文件
- 不允许描述或执行任何第三方工具调用语法
- 不允许出现多模型分流、并行、重试、轮询、后台任务等待等执行控制逻辑
- 不允许保存或交接任何会话 ID

## 执行工作流

**规划任务**：$ARGUMENTS

### 🔍 Phase 1：上下文全量检索

`[模式：研究]`

#### 1.1 Prompt 增强（必须首先执行）

**Prompt 增强**：分析 $ARGUMENTS 的意图、缺失信息、隐含假设，补全为结构化需求（明确目标、技术约束、范围边界、验收标准），**用增强结果替代原始 $ARGUMENTS** 用于后续所有阶段。

#### 1.2 上下文检索

- 使用 `ace-tool` 基于增强后的需求进行语义检索
- 使用自然语言构建查询（Where/What/How）
- **禁止基于假设回答**
- 若 `ace-tool` 不可用：回退到 Glob + Grep 进行文件发现与关键符号定位

#### 1.3 完整性检查

- 必须获取相关类、函数、变量的**完整定义与签名**
- 若上下文不足，触发**递归检索**
- 优先输出：入口文件 + 行号 + 关键符号名；必要时补充最小代码片段（仅用于消除歧义）

#### 1.4 需求对齐

- 若需求仍有模糊空间，**必须**向用户输出引导性问题列表
- 直至需求边界清晰（无遗漏、无冗余）

### 💡 Phase 2：单模型分析与规划

`[模式：分析]`

#### 2.1 单模型分析

将**原始需求**（不带预设观点）交给单一模型分析：

- ROLE_FILE: `./analyzer.md`
- 关注：技术可行性、架构影响、性能考量、潜在风险、交互影响
- OUTPUT: 多角度解决方案 + 优劣势分析

#### 2.2 生成计划草案

在分析结果基础上，由单一模型生成“计划草案”（**不允许**修改任何产品文件）：

- ROLE_FILE: `./architect.md`
- OUTPUT: Step-by-step plan + pseudo-code（重点：数据流、边界条件、交互影响、测试策略）

#### 2.3 创建 plan.md（Claude 最终版）

综合单模型分析，生成并写入 **`.claude/plan/<功能名>.md` 实施计划**：

```markdown
## 📋 实施计划：<任务名称>

### 任务类型
- [ ] 前端
- [ ] 后端
- [ ] 全栈

### 技术方案
<基于单模型分析的最优方案>

### 实施步骤
1. <步骤 1> - 预期产物
2. <步骤 2> - 预期产物
...

### 关键文件
| 文件 | 操作 | 说明 |
|------|------|------|
| path/to/file.ts:L10-L50 | 修改 | 描述 |

### 风险与缓解
| 风险 | 缓解措施 |
|------|----------|
```

### ⛔ Phase 2 结束：计划交付（非执行）

**`/ac-plan` 的职责到此结束，必须执行以下动作**：

1. 创建并写入 `.claude/plan/<功能名>.md`（功能名从需求中提取，如 `user-auth`、`payment-module` 等）
2. 向用户展示完整实施计划（含伪代码）
3. 以**加粗文本**输出提示（必须使用实际保存的文件路径）：

   ---
   **📋 计划已生成并保存至 `.claude/plan/实际功能名.md`**

   **请审查上述计划，您可以：**
   - 🔧 **修改计划**：告诉我需要调整的部分，我会更新计划
   - ▶️ **执行计划**：复制以下命令到新会话执行

   ```
   /ac-execute .claude/plan/实际功能名.md
   ```
   ---

   **⚠️ 注意**：上面的 `实际功能名.md` 必须替换为你实际保存的文件名！

4. **立即终止当前回复**（Stop here. No more tool calls.）

**⚠️ 绝对禁止**：
- ❌ 问用户 "Y/N" 然后自动执行（执行是 `/ac-execute` 的职责）
- ❌ 对产品代码进行任何写操作
- ❌ 自动调用 `/ac-execute` 或任何实施动作
- ❌ 在用户未明确要求修改时继续触发模型调用

---

## 计划保存

规划完成后，将计划保存至：

- **首次规划**：`.claude/plan/<功能名>.md`
- **迭代版本**：`.claude/plan/<功能名>-v2.md`、`.claude/plan/<功能名>-v3.md`...

计划文件写入应在向用户展示计划前完成。

---

## 计划修改流程

如果用户要求修改计划：

1. 根据用户反馈调整计划内容
2. 更新 `.claude/plan/<功能名>.md` 文件
3. 重新展示修改后的计划
4. 再次提示用户审查或执行

---

## 后续步骤

用户审查满意后，**手动**执行：

```bash
/ac-execute .claude/plan/<功能名>.md
```

---

## 关键规则

1. **仅规划不实施** – 本命令不执行任何代码变更
2. **只创建 plan.md** – 产物仅限 `.claude/plan/*.md` 计划文件
3. **不问 Y/N** – 只展示计划，让用户决定下一步
4. 外部模型对文件系统**零写入权限**
5. **禁止会话交接** – 计划中不得包含任何 `SESSION_ID` 或等价会话标识
