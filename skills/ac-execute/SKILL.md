# Execute - 主线程实施 + Subagents 辅助

$ARGUMENTS

---

## 核心协议

- **语言协议**：与工具交互用**英语**，与用户交互用**中文**
- **实施主权**：所有产品代码修改由 Claude 主线程直接执行
- **按需使用 subagents**：仅在上下文检索范围大、实现涉及多个模块、或交付前需要额外只读自审时，使用 `Agent` 工具启动 subagents 辅助
- **只读辅助**：subagents 只负责检索、分析、自审，不直接编辑代码、不写计划文件、不替代主线程交付
- **最小变更**：只修改计划范围内的必要代码
- **前置条件**：优先基于 `/ac-plan` 生成且已批准的计划执行；若缺失计划或计划信息不全，先向用户确认

---

## 执行约束

**工作目录**：
- `{{WORKDIR}}`：使用当前工作目录的绝对路径作为上下文检索与代码修改根目录
- 如果用户通过 `/add-dir` 添加了多个工作区，先用 Glob/Grep 确定任务相关的工作区
- 如果无法确定，用 `AskUserQuestion` 询问用户选择目标工作区

**subagents 使用条件**：
- 适用：关键路径不明确、需要并行摸排多个模块、改动完成后需要额外只读审查
- 优先：`Explore` 用于上下文检索；交付前的附加审查也应优先使用只读 subagent
- 不适用：单文件小改动、路径明确、主线程一次阅读即可完成的简单任务

**重要**：
- 不允许使用 Bash 调用外部模型、生成原型或执行额外编排
- 不允许保存、引用或交接任何外部会话标识
- subagents 不得直接实施代码修改，禁止多代理并行写同一任务
- 已委托给 subagents 的检索或审查内容，主线程不要重复执行
- 仅在本地验证时，按需运行最小相关范围的 lint / typecheck / tests
- 若信息不足，必须先提问再继续执行

## 执行工作流

**执行任务**：$ARGUMENTS

### 📖 Phase 0：读取计划

`[模式：准备]`

1. **识别输入类型**：
   - 计划文件路径（如 `.claude/plan/xxx.md`）
   - 直接的任务描述
2. **读取计划内容**：
   - 若提供了计划文件路径，读取并解析
   - 提取：任务类型、实施步骤、关键文件、风险与验证要求
3. **执行前确认**：
   - 若输入为直接任务描述：先确认是否已有已批准的计划
   - 若计划中缺失关键文件、验收标准或范围边界：先向用户补充确认
   - 若无法确认用户是否已同意执行：必须先询问，再进入下一阶段

### 🔎 Phase 1：上下文检索

`[模式：检索]`

1. 优先读取计划中列出的关键文件
2. 若计划未给出完整路径或实现涉及多个模块：
   - 由主线程按需使用 `Agent` 工具并行启动 subagents
   - 可按文件组、模块或关注点拆分检索任务
3. 补充读取实施所需的入口文件、依赖模块、相关类型定义
4. 若发现计划与当前代码明显不一致，暂停并向用户确认

**检索完成后**：
- 整理关键上下文
- 确认已具备实施所需信息
- 进入 Phase 2

---

### ⚡ Phase 2：主线程实施

`[模式：实施]`

1. 基于计划与上下文**直接实施**
2. 先读取目标文件，理解现有实现方式
3. 由主线程使用 Edit / Write 工具执行实际修改
4. 遵循项目现有代码风格与结构
5. 仅修改必要代码，避免顺手重构和超范围变更
6. 若实施过程中发现计划需要变更，先暂停并向用户确认

---

### ✅ Phase 3：验证

`[模式：验证]`

1. 优先运行最小相关范围的 lint / typecheck / tests
2. 若验证失败：先修复回归问题，再继续交付
3. 若项目缺少自动化检查：给出最小手动验证步骤
4. 未验证通过前，不进入最终交付

---

### 🧪 Phase 4：主线程自审与交付

`[模式：审查]`

#### 4.1 自审

- 检查逻辑正确性
- 检查边界条件与副作用
- 检查安全性与输入边界
- 检查是否存在超范围修改
- 检查是否与计划目标一致

#### 4.2 subagents 附加审查（复杂场景按需启用）

当改动跨多个模块、验证成本高、或存在集成风险时：

1. 使用只读 subagents 并行审查不同关注点
2. 典型拆分方式：
   - **正确性与回归风险**
   - **可维护性与集成一致性**
3. 主线程汇总审查结果，决定是否补充修复

#### 4.3 整合修复

1. 根据主线程与 subagents 审查结果执行必要修复
2. 修复后重复最小验证
3. 风险可接受后再交付

#### 4.4 交付确认

向用户报告：

```markdown
## ✅ 执行完成

### 变更摘要
| 文件 | 操作 | 说明 |
|------|------|------|
| path/to/file.ts | 修改 | 描述 |

### 验证结果
- 已执行：<lint / typecheck / tests / 手动验证>
- 结果：<通过 / 失败后已修复 / 无自动化检查>

### 后续建议
1. [ ] <建议的测试步骤>
2. [ ] <建议的验证步骤>
```

---

## 关键规则

1. **主线程实施** – 产品代码只由当前会话主线程修改
2. **subagents 只做辅助检索/审查** – 按需使用，不默认开启，不直接改代码
3. **最小变更** – 仅修改必要代码，不引入额外副作用
4. **先验证后交付** – 能运行检查就先运行检查
5. **信息不全先确认** – 缺少计划、关键文件或验收标准时必须先问用户
