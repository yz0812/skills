# Execute - 单模型执行

$ARGUMENTS

---

## 核心协议

- **语言协议**：与工具交互用**英语**，与用户交互用**中文**
- **单模型模式**：分析、实施、审查均由当前会话内单模型完成
- **内部执行**：不调用额外 Bash 编排外部模型，不复用外部会话，不依赖后台任务
- **代码主权**：所有文件修改由 Claude 直接执行
- **最小变更**：只修改计划范围内的必要代码
- **前置条件**：优先基于 `/ac-plan` 生成的计划执行；若缺失计划或计划信息不全，先向用户确认

---

## 执行约束

**工作目录**：
- `{{WORKDIR}}`：使用当前工作目录的绝对路径作为上下文检索与代码修改根目录
- 如果用户通过 `/add-dir` 添加了多个工作区，先用 Glob/Grep 确定任务相关的工作区
- 如果无法确定，用 `AskUserQuestion` 询问用户选择目标工作区

**重要**：
- 不允许出现 Codex/Gemini、多模型分流、会话复用、并行执行等描述
- 不允许使用 Bash 调用外部模型、生成原型或执行额外编排
- 不允许保存或交接任何 `SESSION_ID`
- 仅在本地验证时，按需运行最小相关范围的 lint / typecheck / tests
- 若信息不足，必须先提问再继续执行

---

## 执行工作流

**执行任务**：$ARGUMENTS

### 📖 Phase 0：读取计划

`[模式：准备]`

1. **识别输入类型**：
   - 计划文件路径（如 `.claude/plan/xxx.md`）
   - 直接的任务描述

2. **读取计划内容**：
   - 若提供了计划文件路径，读取并解析
   - 提取：任务类型、实施步骤、关键文件、风险与验证要求

3. **执行前确认**：
   - 若输入为直接任务描述：先确认是否已有已批准的计划
   - 若计划中缺失关键文件、验收标准或范围边界：先向用户补充确认
   - 若无法确认用户是否已同意执行：必须先询问，再进入下一阶段

4. **任务类型判断**：

   | 任务类型 | 判断依据 |
   |----------|----------|
   | 前端 | 页面、组件、UI、样式、布局 |
   | 后端 | API、接口、数据库、逻辑、算法 |
   | 全栈 | 同时包含前后端改动 |

---

### 🔍 Phase 1：上下文检索

`[模式：检索]`

- 优先读取计划中列出的关键文件
- 若计划未给出完整路径或相关符号，使用 `ace-tool`、Glob、Grep 补齐上下文
- 补充读取实施所需的入口文件、依赖模块、相关类型定义
- 若发现计划与当前代码明显不一致，暂停并向用户确认

**检索完成后**：
- 整理关键上下文
- 确认已具备实施所需信息
- 进入 Phase 2

---

### ⚡ Phase 2：单模型实施

`[模式：实施]`

1. 基于计划与上下文**直接实施**，不生成外部原型或 Unified Diff 草稿
2. 先读取目标文件，理解现有实现方式
3. 使用 Edit / Write 工具执行实际修改
4. 遵循项目现有代码风格与结构
5. 仅修改必要代码，避免顺手重构和超范围变更
6. 若实施过程中发现计划需要变更，先暂停并向用户确认

---

### ✅ Phase 3：自检验证

`[模式：验证]`

1. 优先运行最小相关范围的 lint / typecheck / tests
2. 若验证失败：先修复回归问题，再继续交付
3. 若项目缺少自动化检查：给出最小手动验证步骤
4. 未验证通过前，不进入最终交付

---

### 🧪 Phase 4：单模型审查与交付

`[模式：审查]`

#### 4.1 自审

- 检查逻辑正确性
- 检查边界条件与副作用
- 检查安全性与输入边界
- 检查是否存在超范围修改
- 检查是否与计划目标一致

#### 4.2 整合修复

1. 根据自审结果执行必要修复
2. 修复后重复最小验证
3. 风险可接受后再交付

#### 4.3 交付确认

向用户报告：

```markdown
## ✅ 执行完成

### 变更摘要
| 文件 | 操作 | 说明 |
|------|------|------|
| path/to/file.ts | 修改 | 描述 |

### 验证结果
- 已执行：<lint / typecheck / tests / 手动验证>
- 结果：<通过 / 失败后已修复 / 无自动化检查>

### 后续建议
1. [ ] <建议的测试步骤>
2. [ ] <建议的验证步骤>
```

---

## 关键规则

1. **单模型内部执行** – 不调用外部模型，不做额外 Bash 编排
2. **直接实施** – 不依赖原型 Diff，不做多阶段会话拼接
3. **最小变更** – 仅修改必要代码，不引入额外副作用
4. **先验证后交付** – 能运行检查就先运行检查
5. **信息不全先确认** – 缺少计划、关键文件或验收标准时必须先问用户
