## React 开发规则

本文件补充 React 专属规则；若与项目级 `CLAUDE.md` 冲突，以项目级规则为准。

## 基本原则

1. 遵循 React 官方 **Thinking in React** 思路：先拆组件、再定数据流、最后加交互。
2. 遵循 **Rules of React** 与 **Rules of Hooks**。
3. 组件和 Hooks 必须保持可预测、可推理、可维护。
4. 状态最小化，数据单向流动。

## 组件设计规则

1. 先按职责拆组件，不按文件大小机械拆分。
2. 组件应关注单一问题，过大时再自然拆解。
3. 优先从静态版本开始，再逐步加状态与交互。
4. 组件层级尽量贴合 UI 结构和数据结构。

## 状态管理规则

1. 状态只保留**最小但完整**的可变数据。
2. 能由 props 或现有 state 推导出的值，不要重复存 state。
3. 状态尽量放在最近公共父组件。
4. 优先保持单向数据流，不搞隐式双向绑定。
5. 修改状态必须通过 setter，不直接改 state 对象或数组。

## React 规则（强约束）

1. 组件和 Hooks 必须纯：相同输入应产生相同输出。
2. render 中禁止副作用。
3. render 中不要调用非幂等逻辑，如 `new Date()`、`Math.random()` 直接决定 UI。
4. 副作用应放在事件处理器或 Effect 中。
5. props、state、传给 Hook 的参数都视为不可变快照。
6. JSX 使用过的值之后不要再原地修改。

## Hooks 规则（强约束）

1. Hooks 只能在 React 函数组件或自定义 Hook 中调用。
2. Hooks 只能在顶层调用。
3. 禁止在条件、循环、嵌套函数、事件处理器、`try/catch/finally` 中调用 Hooks。
4. 不要把 Hook 当普通值到处传递。
5. 不要直接把组件函数当普通函数调用。

## 数据与更新规则

1. 父组件负责持有状态，子组件通过 props 接收数据和回调。
2. 子组件要修改状态，通过上层传入的回调显式上推。
3. 本地临时变量允许局部 mutation，但不能污染 render 外部值。
4. 对数组和对象更新，优先返回新值，不做原地修改。

## 安全规则

1. React 默认 JSX 转义并不代表可以忽视 XSS。
2. `dangerouslySetInnerHTML` 默认视为高风险，非必要不用。
3. 必须使用时，先说明来源可信性与清洗策略。
4. 来自后端、用户输入、URL、富文本的内容都按不可信处理。

## 工程实践

1. 建议启用 `StrictMode` 与 React ESLint 规则。
2. 优先沿用项目现有组件组织、状态方案、路由和样式体系。
3. 不因为个人偏好，随意把简单局部状态升级成全局状态方案。
4. 不为小需求引入额外状态管理库。

## 实现倾向

✅ 先静态组件，再交互

✅ 最小 state，能推导就不存

✅ 单向数据流，状态上移到最近公共父组件

✅ 用事件处理器 / Effect 处理副作用

✅ 保持 props / state / hook 参数不可变

## 严禁事项

* ❌ 在 render 中做副作用
* ❌ 直接修改 props 或 state
* ❌ 条件/循环里调用 Hooks
* ❌ 组件函数当普通函数直接调用
* ❌ 随意使用 `dangerouslySetInnerHTML`
