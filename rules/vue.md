## Vue 开发规则

本文件补充 Vue 专属规则；若与项目级 `CLAUDE.md` 冲突，以项目级规则为准。

## 基本原则

1. 优先遵循 Vue 官方 Style Guide 的 Priority A / B 规则。
2. 组件命名、props、模板表达式、样式作用域要保持一致。
3. 优先清晰的数据流和模板可读性，不炫技。
4. 没有明确收益时，不要把简单逻辑塞进复杂指令组合和魔法写法。

## 组件设计规则

1. 单文件组件项目中，默认一个组件一个文件。
2. 用户自定义组件名默认使用**多单词命名**，根组件 `App` 可例外。
3. 组件文件名保持统一风格：要么 PascalCase，要么 kebab-case，不混用。
4. 基础展示型组件建议统一前缀，如 `Base` / `App` / `V`。
5. 强耦合子组件名称带上父组件前缀，便于组织和检索。

## Props 规则

1. 提交代码中的 props 定义应尽可能详细，至少声明类型。
2. 能声明 `required`、默认值、validator 时优先明确声明。
3. 不要长期保留仅适合原型阶段的 `props: ['x']` 简写。
4. prop 声明用 camelCase；在 in-DOM 模板中使用 kebab-case。

## 模板规则

1. 模板中只保留简单表达式；复杂逻辑提到 `computed` 或方法中。
2. `v-for` 必须配稳定 `key`，优先使用业务唯一 id。
3. **禁止在同一元素上同时使用 `v-if` 和 `v-for`**。
4. 需要过滤列表时，优先计算出新列表；需要整体隐藏时，把 `v-if` 挪到容器层。
5. 多属性元素默认拆成多行，一行一个属性，提升可读性。
6. 指令缩写风格保持一致：要么统一缩写，要么统一全写。
7. 非空 HTML 属性值保持带引号。

## 组件与模板命名规则

1. SFC / string template 中组件名优先 PascalCase。
2. in-DOM 模板中组件名使用 kebab-case。
3. JS/TS 中组件引用优先 PascalCase。
4. 命名优先完整单词，不要滥用缩写。

## 计算属性与状态规则

1. 复杂 computed 要拆分成多个简单 computed。
2. computed 命名要表达结果语义，不要模糊命名。
3. 可推导值不要重复存状态。
4. 数据流要显式，避免模板里堆叠条件分支和链式变换。

## 样式规则

1. 除顶层 `App` 和布局组件外，其余组件样式默认应作用域隔离。
2. 可使用 `scoped`、CSS Modules、BEM 等方式，但项目内保持一致。
3. 组件库更倾向使用类名策略而非 `scoped`，以利覆盖和复用。
4. 避免通用类名污染全局，如 `button`、`icon`、`btn` 无边界扩散。

## 实现倾向

✅ 一个组件一个文件

✅ 多单词组件名

✅ 明确 props 类型

✅ `v-for` 带稳定 key

✅ 复杂模板逻辑上移到 computed / methods

✅ 样式作用域清晰

## 严禁事项

* ❌ 单词组件名泛滥
* ❌ `v-for` 不写 key
* ❌ `v-if` 和 `v-for` 同元素混用
* ❌ 模板中堆复杂表达式
* ❌ 全局样式无边界污染
