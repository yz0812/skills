## Node.js 开发规则

本文件补充 Node.js 专属规则；若与项目级 `CLAUDE.md` 冲突，以项目级规则为准。

## 基本原则

1. Node.js 应用默认按服务端高权限进程处理，不按“普通脚本”放松要求。
2. 依赖安全、输入校验、资源限制、运行边界同等重要。
3. 优先使用项目现有包管理、构建、运行方式，不擅自切生态。
4. 能通过平台能力和配置约束风险，就不要只靠代码自觉。

## 依赖与供应链安全

1. 依赖必须锁版本，优先依赖 lockfile 保证直接和传递依赖可复现。
2. 新增依赖前先确认：是否活跃维护、是否必要、是否已有同类依赖。
3. 对 typosquatting、恶意包、维护者被接管、lockfile poisoning 保持警惕。
4. 安装依赖时，谨慎执行 install/postinstall 脚本；必要时考虑 `--ignore-scripts`。
5. 定期运行依赖漏洞检查，如 `npm audit` 或项目既有安全扫描流程。
6. 发布前用 `npm publish --dry-run` 检查会被发布的文件。

## 包与发布规则

1. 严格控制发布内容，使用 `files`、`.npmignore`、`.gitignore` 管理暴露面。
2. 严禁把密钥、证书、环境变量模板中的敏感值、调试文件发布到 npm。
3. 发布配置与源码仓库内容不完全等价，必须检查实际发布产物。
4. 不为小工具随意发布公开包；先确认是否真有复用价值。

## 服务端安全

1. 所有外部输入都按不可信处理：HTTP 请求、消息队列、CLI 参数、环境变量、文件内容。
2. 默认防注入：SQL、命令、路径、模板、序列化、正则 DoS。
3. 避免使用危险动态执行，如 `eval()`，除非明确必要且边界可控。
4. 不要开启 `insecureHTTPParser`。
5. 调试能力如 `--inspect` 禁止跑在生产暴露面上。

## HTTP 与资源限制

1. 正确配置 `headersTimeout`、`requestTimeout`、`keepAliveTimeout` 等超时。
2. 限制 socket、连接数、请求体大小、上传大小、并发度，防止 DoS / Slowloris。
3. 能放反向代理的场景，优先让反向代理承担限流、缓存、黑名单等职责。
4. 对流式处理、文件上传、压缩解压、大 JSON 解析要考虑内存压力。

## 权限与运行边界

1. 能启用更小运行权限就不要默认全开。
2. 文件系统、网络、子进程、原生扩展访问要谨慎。
3. 若使用 Node.js Permission Model，按最小授权原则配置。
4. 生产环境避免与不可信工作负载共享机器。

## 代码实现规则

1. 优先显式、稳定、易审计的写法，不炫技。
2. 对象合并、深拷贝、配置合并要防 prototype pollution。
3. 判断对象自有属性优先 `Object.hasOwn()`。
4. 需要无原型对象时优先 `Object.create(null)`。
5. 避免不安全递归 merge。
6. 不随意 monkey patch 全局对象或原型。

## 敏感数据处理

1. 机密信息不能写死在代码里。
2. 比较敏感值时优先常量时间比较能力，如 `crypto.timingSafeEqual`。
3. 密码处理、令牌处理优先成熟标准库，不自己手写密码学逻辑。
4. 日志中禁止泄露 token、密钥、用户敏感信息。

## 测试与验证

1. 修 bug 优先补复现测试。
2. 涉及超时、并发、流、权限、依赖安装、发布内容时，必须做专项验证。
3. 能跑最小相关测试，就不要默认全量跑。
4. 若项目已有 lint / typecheck / test / security scan，优先沿用。

## 实现倾向

✅ 锁定依赖 + 审查新增包

✅ 用显式输入校验和参数化接口代替拼接式危险写法

✅ 用超时、限流、代理、权限模型降低攻击面

✅ 发布前核对实际 npm 包内容

## 严禁事项

* ❌ 未确认前随意新增高风险依赖
* ❌ 把 `--inspect`、调试接口暴露到生产
* ❌ 把 secrets 打进 npm 包或仓库产物
* ❌ 依赖 install 脚本做不可控系统副作用
* ❌ 直接信任外部 JSON、对象合并结果、用户输入
