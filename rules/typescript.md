## TypeScript 开发规则

本文件补充 TypeScript 专属规则；若与项目级 `CLAUDE.md` 冲突，以项目级规则为准。

## 基本原则

1. 默认开启并坚持 **`strict: true`**。
2. 类型系统是约束和建模工具，不是装饰品。
3. 优先类型正确、边界清晰、模块一致，不滥用断言逃避问题。
4. 能让编译器发现问题，就不要把错误留到运行时。

## tsconfig 规则

1. 新增或调整 TS 项目配置时，优先从运行环境出发，而不是拍脑袋套模板。
2. App 项目按环境拆 tsconfig：浏览器、Node、worker、测试代码尽量分开。
3. Bundler 项目优先结合官方建议选择 `module` / `moduleResolution`。
4. Node 运行时项目优先按 `nodenext` / 对应 Node 模块语义配置。
5. Library 项目优先采用更严格配置，避免把宽松类型问题传播给使用者。

## 类型设计规则

1. 优先精确类型，不要动不动 `any`。
2. 能用联合类型、字面量类型、枚举语义表达清楚的，就不要偷懒写字符串随便传。
3. 公共 API 类型优先稳定、显式、可组合。
4. 类型命名必须体现业务语义，不要堆 `Data`、`Info`、`Obj` 这类废名。
5. 区分输入类型、内部类型、对外暴露类型，不要图省事共用一套脏类型。

## 严格模式规则

1. `strict` 默认不开例外，除非项目现状明确不允许。
2. 新增代码不能靠关闭 strict 子选项逃避问题。
3. `strictNullChecks` 语义必须认真处理，不要用非空断言 `!` 到处糊过去。
4. `noImplicitAny` 语义必须正视，出现 `any` 要说明原因。

## 模块与导入规则

1. 模块配置必须与实际运行环境一致。
2. 不要混乱使用 ESM / CJS 导致编译过了、运行炸掉。
3. 导出结构优先稳定可读，减少模糊 re-export 污染。
4. 对库项目，优先选择能兼容消费者环境的模块配置。
5. 不为省事写只在 bundler 能跑、Node 里会炸的导入方式。

## 实现规则

1. 能通过类型建模解决的问题，不要留给运行时兜底。
2. 不要滥用 `as`、双重断言、`unknown as X`、`any as X`。
3. 断言只能用于你已明确验证过、编译器无法表达的边界情况。
4. 优先缩小类型（narrowing），而不是粗暴覆盖类型。
5. 公共函数签名先设计清楚再写实现。

## 库与应用差异

1. App 项目以“匹配运行环境”为主。
2. Library 项目以“对消费者最稳妥”为主。
3. Library 项目建议显式声明：`declaration`、`rootDir`、`outDir` 等。
4. 发布库时，避免产出和声明文件之间的模块语义不一致。

## 工程实践

1. 若项目已有 ESLint / typecheck / build 流程，必须沿用。
2. 修改 tsconfig 前先说明影响：构建、运行环境、声明文件、兼容性。
3. 不随意切换模块系统、打包方式、路径别名策略。
4. 类型问题优先修根因，不要在业务代码四处补断言。

## 实现倾向

✅ `strict: true`

✅ 精确类型优先于 `any`

✅ 按运行环境配置模块系统

✅ 用类型缩小代替暴力断言

✅ 公共类型与内部类型分层

## 严禁事项

* ❌ 随意关闭 strict 子项来掩盖问题
* ❌ 用 `any` 当默认解决方案
* ❌ 大量 `as` / 非空断言把类型系统架空
* ❌ bundler / Node 模块配置混乱
* ❌ 发布产物与声明文件语义不一致
