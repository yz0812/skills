## Database / SQL 开发规则

本文件补充数据库开发规则；若与项目级 `CLAUDE.md` 冲突，以项目级规则为准。

## 基本原则

1. 默认按高风险边界处理数据库操作。
2. 优先正确性、原子性、可回滚性，再谈“写得快”。
3. 默认最小权限、参数化、显式事务。
4. 不把数据库当字符串拼接器使用。

## SQL 安全规则

1. **首选 Prepared Statements / 参数化查询**。
2. **禁止把用户输入直接拼接进 SQL**。
3. 表名、列名、排序方向等无法绑定的位置，必须走 allow-list 映射。
4. 逃逸用户输入不是首选方案，不能当主要防线。
5. Stored Procedure 只有在安全实现时才等价安全；内部若继续拼动态 SQL，一样危险。

## 输入校验规则

1. 所有外部输入都必须先做类型、范围、语义校验。
2. 能转成布尔、枚举、数字、日期的输入，先转类型再参与查询。
3. 对排序字段、表名、列名、分页字段，必须做白名单映射。
4. 不把“校验通过”误当成“可以安全拼接 SQL”。

## 权限规则

1. 应用账号遵循最小权限原则。
2. 只读场景只给读权限，禁止图省事给 DBA / owner 权限。
3. 不同应用、不同职责尽量使用不同数据库账号。
4. 能用 view 收窄读取范围时，优先考虑 view。
5. 不要让应用账号拥有不必要的 DDL、删除、管理权限。

## 事务规则

1. 涉及多步修改且必须一起成功时，必须放进事务。
2. 事务的目标是 **all-or-nothing**，中间状态不应对外可见。
3. 出错时明确 `ROLLBACK`，成功时明确 `COMMIT`。
4. 能缩小事务范围就缩小，避免长事务占锁。
5. 需要局部回退时，考虑 savepoint，而不是靠手工补偿 SQL 硬凑。

## 查询与建模规则

1. 先沿用项目现有 ORM / Mapper / SQL 风格，不混多套范式。
2. 能在数据库高效完成过滤、分页、聚合，就不要全拉回应用内存处理。
3. 关注索引命中、分页成本、锁竞争、N+1、批量写入成本。
4. 修改 schema、索引、约束前，必须评估兼容性与回归风险。
5. 对大表操作、批量更新、清理脚本保持保守。

## 变更规则

1. 任何 schema / migration / 索引变更都属于高风险改动。
2. 修改前先说明影响：数据量、锁、回滚方式、兼容性、停机风险。
3. 配置、migration、DDL 变更不要偷偷做，必须先报备。
4. 对生产数据相关操作，默认先让用户确认。

## 调试与审计

1. 日志中不要输出敏感 SQL 参数值。
2. 排查问题时可记录 SQL 模板、耗时、行数，但敏感值要脱敏。
3. 优先分析慢查询、执行计划、索引使用，而不是拍脑袋优化。
4. 不为了排查方便长期保留危险调试开关。

## 实现倾向

✅ 参数化查询

✅ 表名/列名/排序字段白名单映射

✅ 事务包裹多步原子更新

✅ 最小权限数据库账号

✅ 用 view、索引、执行计划做精细化控制

## 严禁事项

* ❌ 字符串拼接 SQL + 用户输入
* ❌ 用管理员账号跑应用查询
* ❌ 不加事务执行多步关键写操作
* ❌ 在 stored procedure 里继续裸拼动态 SQL
* ❌ 未确认前擅自执行高风险 DDL / migration / 批量更新
